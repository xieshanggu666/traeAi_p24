## 第三轮修复（回复瓶子无权限）

### 问题5：捞瓶子回复消息时报错"瓶子不存在或无权限查看"

**问题描述**：用户捞到瓶子后，在聊天页发送回复时，调用 `GET /api/bottle/:bottleId` 接口返回 `瓶子不存在或无权限查看` 错误。

**错误位置**：`server/routes/bottle.js:1003`

```javascript
if (rows.length === 0) {
  return res.status(404).json(generateResponse(false, null, '瓶子不存在或无权限查看'));
}
```

**根本原因**：`GET /:bottleId` 接口的查询条件要求 `(b.sender_id = ? OR b.picker_id = ?)`，即用户必须是发送者或瓶子上记录的捞取者才能查看详情。但是：

1. **`/pick` 捞瓶子接口**只更新了 `pick_count`（捞取次数）和插入了 `bottle_pick_records` 捞取记录表，**没有把捞取者ID写入 `bottles.picker_id` 字段**，也没有设置 `picked_at` 捞取时间
2. **`/reply` 回复接口**也没有把 `picker_id` 写入 `bottles` 表
3. 导致捞取者（非发送者）访问 `GET /:bottleId` 时，`picker_id` 为 `NULL`，条件 `b.picker_id = userId` 永远不成立，`sender_id = userId` 也不成立，所以返回无权限

**完整数据流问题**：

| 步骤 | 接口 | 实际行为 | 期望行为 |
|------|------|---------|---------|
| 1 | POST /pick | pick_count+1, 插入pick_records | 额外：写入picker_id、picked_at、status='picked' |
| 2 | POST /reply | status='replied' | 额外：如果picker_id为空，补齐picker_id和picked_at |
| 3 | GET /:bottleId | 检查 sender_id OR picker_id | 正确（但picker_id为空导致失败） |

**修复方案**：

### 修复1：`/pick` 接口写入 picker_id
`server/routes/bottle.js:461-464`
```javascript
// 修改前
UPDATE bottles SET pick_count = pick_count + 1 WHERE id = ?

// 修改后
UPDATE bottles SET pick_count = pick_count + 1, 
       picker_id = ?, picked_at = NOW(), status = 'picked' 
WHERE id = ?
```
同时把状态从 `floating` 改为 `picked`，防止再次被别人捞到。

### 修复2：`/reply` 接口增加权限校验+兼容旧数据
`server/routes/bottle.js:562-589`
- 校验：如果瓶子已有 `picker_id`，必须是当前回复用户
- 校验：如果没有 `picker_id`，检查当前用户是否在 `bottle_pick_records` 捞取过
- 写入：更新状态为 `replied` 时，用 `IFNULL(picker_id, ?)` 补齐缺失的 `picker_id` 和 `picked_at`

```javascript
UPDATE bottles SET status = 'replied', 
       picker_id = IFNULL(picker_id, ?), 
       picked_at = IFNULL(picked_at, NOW()) 
WHERE id = ?
```

**状态流转（修复后）**：
```
扔瓶子 → floating → 捞瓶子 → picked → 回复消息 → replied
                          ↓
                      扔回海里 → floating
```

## 第二轮修复

### 问题3：皮肤列表返回15条数据，后面7条无样式

**问题描述**：商店皮肤接口返回了15条皮肤数据，但只有前8条有完整样式，后面7条因缺少渐变色、表情等字段导致页面无法正常渲染。

**根本原因**：数据库中残留了之前版本的旧皮肤记录（如"心有所属"、"绿意盎然"等），这些记录的ID不在当前`SKINS`数组中。迁移脚本的更新逻辑仅处理SKINS数组中定义的8款皮肤，旧记录不会被更新也不会被删除。

**修复方案**：在`server/config/migrateSkins.js:244-255`添加清理逻辑，迁移时自动删除所有ID不在SKINS数组中的旧记录。

**验证结果**：重启日志显示 `已清理 8 条旧皮肤记录`

### 问题4：购买皮肤后首页漂流瓶样式没有变化

**问题描述**：用户购买并激活皮肤后，首页的漂流瓶图标仍显示默认的🍾，没有任何皮肤效果。

**根本原因**：`client/src/views/Home.vue`中瓶子图标是硬编码的，没有获取用户当前使用的皮肤数据。

**修复方案**：在Home.vue中添加皮肤获取和动态显示逻辑：
- 导入`getMySkins` API，在`onMounted`中获取用户激活的皮肤
- 瓶子图标改为动态显示，根据皮肤表情
- 添加皮肤专属光晕背景、边框色投影
- 瓶子下方显示当前皮肤名称标签

**修复位置**：`client/src/views/Home.vue`
- 模板：第24-31行
- 脚本：第72、79、86-110、132-139行
- 样式：第215-238行

## 第一轮修复

### 问题1：皮肤页面中间皮肤未正常显示

**根本原因**：数据库中`bottle_skins`表存在旧的不完整数据，缺少`emoji`、`gradient_from`、`gradient_to`、`border_color`、`rarity`等关键字段。

**修复方案**：修改`server/config/migrateSkins.js`中的皮肤数据插入逻辑，将`INSERT IGNORE`改为先查后更新/插入。

**修复位置**：`server/config/migrateSkins.js:211-242`

### 问题2：购买皮肤报错 Unknown column 'duration'

**根本原因**：数据库中`user_skins`表已存在但为旧版本表结构，缺少`duration`、`price`、`expire_at`、`is_active`等必需字段。

**修复方案**：在`server/config/migrateSkins.js`中添加`user_skins`表的字段修补逻辑，自动检测并添加缺失字段。

**修复位置**：`server/config/migrateSkins.js:176-194`
